02 — Problem Statement

What We Are Solving

India has 500 million+ students. Every major EdTech tool treats them as a homogeneous group. The result: students disengage, fall behind, and lose confidence. Here is the precise problem breakdown.

The Student-Side Failures

  • No Real Personalization: Platforms like Byjus, Unacademy, and Khan Academy serve pre-recorded or semi-adaptive content — not true per-student adaptation. A JEE student who is weak in rotational mechanics but strong in thermodynamics gets the same video as everyone else.
  • Fragmented Tools: Students use Photomath for solving, Notion for notes, YouTube for videos, Anki for flashcards, and Chegg for doubts. No single system unifies all learning needs. Cognitive load from tool-switching kills focus.
  • No Memory of Past Mistakes: Every chatbot conversation starts fresh. If a student made the same integration error 20 times, no AI currently tracks and intervenes on that specific pattern across sessions.
  • Zero Emotional Intelligence: Students stress, burnout, and panic near exams. Existing tools have no mechanism to detect mood and adapt teaching accordingly. A stressed student needs simplification, not harder problems.
  • Passive Learning Dominance: 90% of existing EdTech is watch-and-hope. Active recall, spaced repetition, interleaving, and problem-solving under time pressure are academic gold standards that are rarely implemented adaptively.
  • Weak Assessment: Most platforms use MCQ quizzes that measure memory, not understanding. Students can pass a quiz by guessing. True assessment requires viva-style questioning, step-by-step grading, and pattern analysis.
  • No Career to Learning Bridge: Students study isolated chapters without understanding why. No platform currently connects what a student is learning today to their career goal 5 years from now in a personalized, real-time way.

05 — Environments System

The Five Environments of DRONA AI

An Environment in DRONA AI is not a page, a tab, or a section. It is a complete reconfiguration of the entire interface — the header theme, sidebar content, active agents, available features, color language, and interaction logic all change simultaneously when a student switches environments. Think of it like switching between different apps that share the same underlying brain.

Core Environment Rule: Each environment displays only its own features in the sidebar. The Header is always visible with the Environment Switcher centered. The Logo + Platform Name remain constant. The Profile bar (Avatar, XP, Level) persists across all environments. The Head Agent (Drona) is always available regardless of environment. Only sub-agents change per environment. No feature bleeds into the wrong environment's sidebar — but any feature can be called cross-environment via slash commands.
Environment Switching Architecture

🎨 The Environment Transition Animation

When a student clicks a different Environment in the header, the following happens in sequence over approximately 600 milliseconds: (1) The current sidebar fades out with a 150ms opacity transition. (2) The header pills animate — the current active pill shrinks and desaturates, the new active pill expands with the new environment's accent color. (3) The background and border colors interpolate via CSS custom property transitions from the current environment's palette to the new one. (4) The sidebar rebuilds with the new environment's pages using a staggered slide-in animation at 50ms intervals per item. (5) The main content area cross-fades to the new environment's default page. This transition communicates "you are somewhere different now" without being jarring. The animation respects prefers-reduced-motion — on that setting, transitions happen instantly without animation.

Test Features — Detailed

AI Mock Test Generator

The Test Agent generates full-length mock exams tailored to the student's specific weakness heatmap. Input parameters: Subject(s), Difficulty (Easy/Medium/Hard/Mixed), Duration (30min / 1hr / 2hr / 3hr Full Length), Question Type (MCQ only / MCQ + Numerical + Subjective), Marking Scheme (positive-only / negative marking / step marking). The agent retrieves questions from the Platform PYQ database, generates new questions using the LLM for gap areas, and assembles them into a properly formatted exam interface with timer, section navigation, and flag-for-review functionality. After submission, the Grader Agent evaluates every answer — explaining why wrong answers are wrong, which concept was tested, and what the correct conceptual chain is.

Viva Examiner Simulation

A text-based viva session where the Viva Agent plays the role of a strict examiner — asking questions verbally (text form), judging the student's explanation quality, probing deeper when answers are shallow ("But WHY does that happen?"), and scoring on communication clarity, conceptual accuracy, and completeness. After 10-15 minutes, the Viva Agent gives a structured score and detailed feedback. This is one of the most unique features on DRONA — no competitor currently offers AI-powered viva practice at this level of depth and adaptivity.

Shadow-Simulation — NTA Difficulty Curve Replication

The most advanced test feature. The Shadow Agent analyzes the difficulty distribution pattern of the last 10 years of NTA (National Testing Agency) JEE/NEET exams. It replicates not just question difficulty, but the sequence: typically easy start → medium middle → hard trap questions → time-sink MCQs at the end. During the live mock, it tracks the student's dwell time on each question and adjusts upcoming question presentation order (not content) to simulate the psychological pressure of a real NTA exam. Students who train with Shadow-Simulation are more mentally prepared for actual exam day than those who only take standard mocks.

Handwritten Answer Evaluator

Students photograph or scan handwritten solutions to subjective/numerical questions and upload directly in the Test Environment. The OCR engine (Google Cloud Vision API or Tesseract for math + custom trained) parses the handwritten text and equations. The Grader Agent then evaluates: Is the approach correct? Is each step logically valid? Did the student arrive at the right intermediate values? Is the final answer correct? Step-by-step marks are assigned (like a real board examiner). A detailed markup of the handwritten solution is shown — marking correct steps in green and errors in red with explanations alongside.

Crash Revision Generator

When a student has an exam in 1 day, 3 days, or 7 days, this feature generates a structured crash course. For 1-day crash: Only the highest-probability exam topics (based on PYQ trend analysis) are included, with each topic limited to the most important formula, most common question type, and fastest solving method. For 3-day crash: Covers the top 60% of the syllabus by exam weight with one practice problem per topic. For 7-day crash: Full weak-topic repair plan — each day targets the student's lowest-mastery subjects with a mini-mock at the end of each day to test retention. The Crash Generator is directly linked to the PYQ Analyzer, which predicts this year's likely questions based on 20-year trend analysis.

Penalty System — The Ruthless Rules

⚠️
Philosophy of Penalties: Penalties exist not to punish students but to simulate the real consequence of academic inaction. A student who skips study before JEE faces real-world consequences. The platform mimics this with virtual penalties that sting without being demotivating. The system distinguishes between valid absence (Drona-approved breaks, holidays, emergencies) and unjustified skipping. Only unjustified skipping triggers penalties.
Penalty Tier 1 — Soft Warning (1-2 days skipped)
  • No XP earned for skipped period
  • Daily streak broken and reset to 0
  • Drona sends a firm but motivational message on next login
  • Subject streaks for unattended subjects break
  • Study planner auto-compresses missed content into next 3 days
Penalty Tier 2 — Hard Consequence (3-5 days skipped)
  • Leaderboard rank drops by estimated 20 positions (Rank Agent records the drop)
  • Level does not decrease but "Discipline" stat drops significantly
  • All pending missions expire and new harder missions replace them
  • Topic mastery scores for unrevised topics decay by 5-10%
  • A "Return Assessment" is auto-scheduled — mandatory short test before normal learning resumes
Penalty Tier 3 — Critical Alert (7+ days absent)
  • Parent Report auto-generated and sent (if parent email linked)
  • Avatar enters "Dormant" state with visual degradation indicator on profile
  • Rank drops to bottom 20% if previously in top 50%
  • A full Re-Assessment is recommended by Drona on return
  • All topic mastery scores decay as per forgetting curve for the absent period
  • Drona enters "Catch-Up Mode" and rebuilds a new recovery plan from scratch

Boss Battle — Technical Specification

The Boss Battle is the most social and competitive feature in the Game Environment. It is a real-time, synchronous group test session. Here is exactly how it works:

  1. A student enters the Boss Battle Lobby by selecting a Subject and Topic (e.g., "Organic Chemistry — Named Reactions").
  2. The system waits until at least 10 students have joined the same lobby. Maximum lobby size: 50 students. Wait time shown with live counter.
  3. When minimum threshold is met, a 3-second countdown begins. All students see "3... 2... 1... BATTLE!"
  4. A 20-question test appears simultaneously for all students. Questions are generated fresh by the Test Agent for this session — no repeats from recent sessions.
  5. Timer: 30 minutes for 20 questions. All students see a live timer. No question can be reopened after moving to next (to simulate real exam pressure).
  6. Students can see a live "Rank in Battle" bar on the side showing their position vs. others — updating in real-time as scores are calculated per submitted question.
  7. Final ranking is shown at the end with the top 3 on a podium visual. All participants receive XP based on rank. Winner gets a unique "Battle Champion" badge for that topic.
  8. The entire session data (every student's responses, timing, errors) feeds back into their individual profiles for weakness analysis.

A student who is high in Precision but low in Velocity is an accurate-but-slow learner — Drona addresses this by introducing time-pressure drills. The chart is shown prominently on the Profile page and in the Game Environment's Lobby. Below the radar chart, individual stat bars show the last 30-day trend for each stat.

Every Agent on DRONA AI — Defined

DRONA operates as a multi-agent system where each agent has a defined role, defined rules, a specific prompt architecture, and a clear communication protocol with other agents. Always Active · All Environments · Orchestrator of All Agents

The Head Agent is DRONA's central intelligence. Named after the legendary teacher Dronacharya, this agent acts as a school principal — it does not teach every subject deeply itself but knows enough about all subjects to handle most queries, and delegates to specialist sub-agents when deeper expertise is needed. Core Responsibilities
  • Primary chat interface — handles all student messages first
  • Decides which sub-agent (if any) should handle each query
  • Builds and updates the daily study plan autonomously
  • Monitors real-time session behavior and triggers adaptations

Prompt Engineering Architecture
  • System Prompt Layer 1 (Identity): "You are Drona, the master teacher. You know all subjects. You lead a team of specialist agents..."
  • Layer 2 (Student Profile): Dynamically injected — name, exam target, weaknesses, learning speed, persona preference, current mood signals
  • Layer 3 (Session Context): Today's plan, current topic, recent conversation summary, memory retrieval results
  • Layer 4 (Rules): Answer format rules, depth control, when to route to sub-agents, when to ask pop-up questions, when to suggest breaks

08 — Agent Communication Protocol

How Agents Talk To Each Other

All agents communicate through a structured internal message bus. No agent communication is visible to the student. The student sees one seamless conversation; behind it, agents route queries, share insights, and collaborate to produce the best possible response.

Message Bus Architecture

Every agent interaction follows a strict message protocol. When the Head Agent decides to route a query to a sub-agent, it sends an internal message containing: the student's original query, relevant Memory Mesh entries (pre-retrieved), the student's current profile summary (exam target, current topic, weakness flags), the requested output format (explanation, problem set, quiz, diagram), and any constraints (depth level, persona mode). The sub-agent processes this and returns a structured response. The Head Agent then: (1) Validates the response quality, (2) Adjusts the tone to match the current persona setting, (3) Appends any additional context from the Memory Mesh, and (4) Delivers the final response to the student.

Communication Rules

RULE 1:

The Head Agent is the ONLY agent that communicates directly with the student's frontend chat. All sub-agents communicate only with the Head Agent. The student never sees sub-agent names, routing decisions, or internal messages.

RULE 2:

If two agents produce conflicting recommendations (e.g., Stats Agent says "focus on Physics" but Test Agent says "Chemistry mock is overdue"), the Head Agent resolves the conflict using priority rules: exam proximity overrides long-term planning, weakness severity overrides streak maintenance, and student-initiated requests override all AI suggestions.

RULE 3:

Cross-environment agent communication is asynchronous. The Quest Agent (Game Env) sends daily mission requests to the Head Agent. The Head Agent responds when it has computed the next day's plan. No agent blocks another agent's operation — all inter-agent communication is fire-and-forget with eventual consistency.

RULE 4:

The Memory Agent is a shared service — all agents read from and write to it. No agent "owns" any memory entries. Any agent can create memory entries and any agent can retrieve them. Deduplication and conflict resolution is handled by the Memory Agent's write pipeline, not by individual agents.

10 — Database Architecture

Platform DB vs. User DB — The Dual-Database Model

DRONA operates two completely separate database domains: the Platform Database (shared, admin-curated, read-only for students) and the User Database (private, student-owned, read-write). Both are searchable by AI agents, but with strict priority rules that determine which source takes precedence in different contexts.

Platform Database
The Curriculum Core

Admin-managed, verified, and curated. Students cannot write to this database. All content is tagged with metadata: subject, chapter, class, difficulty, source, question type, year, and exam board.

Contents
  • NCERT Full Text: Classes 10-12, all subjects. Chunked into ~500-word segments, each embedded as a vector for RAG retrieval.
  • PYQ Papers: JEE Main (2004-2024), JEE Advanced (2004-2024), NEET (2004-2024), CET (2010-2024). Each question individually parsed, tagged, and stored with solution.
  • Formula Banks: Every formula in the CBSE/JEE/NEET syllabus. Structured as: formula, variables, derivation source, application examples, common mistakes.
  • Curated Notes: Expert-written notes covering every chapter. Written at two depth levels — "Quick Review" (2-page summary) and "Deep Dive" (10-page comprehensive).
  • Named Reactions DB: 80+ organic chemistry named reactions with mechanisms, conditions, and examples.
  • Mnemonic Bank: Pre-built mnemonics for Biology taxonomy, Chemistry periodic trends, Physics constants, and more.
  • Analogy Library: 500+ verified analogies tagged by concept and interest category (sports, cooking, gaming, movies).
Storage: Structured data in PostgreSQL (metadata, tags, relationships). Content chunks embedded in Pinecone/Weaviate (vector search). Raw files (PDFs, images) in Cloudflare R2.
User Database
The Personal Knowledge Store

Student-owned, private, fully editable. Each student has their own isolated namespace in the vector database. No student can access another student's data. Agents can read from it when the "Use My Database" toggle is ON.

Contents (Student-Created)
  • Notes: Typed notes (markdown), handwritten notes (photos, OCR-processed), and AI-generated summaries saved from chat.
  • Uploads: PDF textbooks, PPTX presentations, DOCX files, images of handwritten solutions.
  • Saved Links: YouTube video links (transcript auto-extracted), website articles (content auto-fetched and summarized).
  • AI Outputs: All AI-generated content that the student saves — flowcharts, mind maps, flashcard decks, audio podcasts, solved solutions.
  • Annotations: Highlights and notes made on PDFs in the integrated reader.
  • Chat Bookmarks: Specific AI responses that the student bookmarks for future reference.
Privacy Rule:

User Database content is NEVER used to train platform-wide models. It is NEVER shared with other students. It is NEVER accessible by admin except under explicit legal requirement with user notification. Isolation is enforced at the database level via namespace separation in the vector DB and Row Level Security in PostgreSQL.

11 — RAG Pipeline & Knowledge Retrieval

How Agents Find & Use Knowledge

RAG (Retrieval-Augmented Generation) is the core mechanism that prevents AI hallucination and ensures curriculum-accurate answers. Instead of relying solely on the LLM's training data, every agent query first retrieves relevant content from the Platform and User databases, then feeds that retrieved context into the LLM prompt alongside the student's question.

RAG Retrieval Flow — Step by Step
1
Student Query Received: "Explain the photoelectric effect and why threshold frequency exists"
2
Embedding Generated: Query text → Gemini Embedding API → 768-dim vector
3
Parallel Search (3 sources):
→ Platform DB Vector Search (Pinecone namespace: "platform")
→ User DB Vector Search (Pinecone namespace: "user_{id}")
→ Memory Mesh Search (Pinecone namespace: "memory_{id}")
4
Re-Ranking & Priority: Results scored by relevance. Platform DB results get 1.2x priority boost. Duplicates deduplicated. Top-5 chunks selected.
5
Context Injection: Top-5 chunks + Memory entries + Student Profile → injected into LLM prompt as context before the student's question.
6
LLM Generates Response: The model answers using retrieved context as its primary source, supplemented by its own knowledge only for gaps.

Data Retrieval Priority Rules

⚠️
Critical Design Decision: When Platform DB and User DB content conflict on the same topic, the Platform DB takes priority for exam-related factual content (formulas, definitions, official syllabus answers). The User DB takes priority for personal context (learning preferences, custom notes, teacher-specific methods). This prevents a student's incorrect personal notes from overriding verified curriculum content during exam preparation.
Scenario Priority Order Reasoning
Exam question solvingPlatform DB → Memory → User DBExam answers must be from verified sources. Student notes are supplementary context only.
Concept explanationPlatform DB → User DB → MemoryCurriculum-aligned explanation first. Student's own notes add personal context.
"Use My Database" toggle ONUser DB → Platform DB → MemoryStudent explicitly wants answers from their own materials. Platform supplements gaps.

Vector Embedding Strategy

Chunking Rules

  • Platform content: Chunk size = 500 tokens, overlap = 100 tokens. Each chunk tagged with: source document, chapter, section, page number.
  • User uploads (PDFs): Chunk size = 400 tokens, overlap = 80 tokens. OCR-processed content is cleaned before embedding.
  • Chat memories: Each memory entry = one fact/insight (typically 50-200 tokens). No chunking needed — memories are atomic units.
  • Re-embedding trigger: If the embedding model is updated, all vectors are re-embedded in a background batch job over 24-48 hours.

Search Configuration

  • Similarity metric: Cosine similarity (normalized dot product). Threshold: 0.72 minimum relevance score.
  • Top-K: Retrieve top 10, re-rank using cross-encoder, select top 5 for injection.
  • Hybrid search: Vector search + BM25 keyword search combined. Weighted 70% vector + 30% keyword for best recall.
  • Namespace isolation: Platform content in shared namespace. Each user gets namespace "user_{uuid}". Memory in "memory_{uuid}". Complete isolation enforced.

16 — Security & Data Privacy

Protecting Student Data — Non-Negotiable

DRONA handles sensitive data: student academic performance, personal notes, learning behaviors, and parent contact information. Security is not an afterthought — it is embedded into every layer of the architecture. India's DPDP Act (Digital Personal Data Protection Act, 2023) and student data protection best practices are the compliance baseline.

🔒 Data Isolation

  • Every student's data is isolated via Supabase Row Level Security (RLS). Even if a backend vulnerability exists, RLS prevents cross-student data access at the database level.
  • Vector DB namespaces are per-student. Memory Mesh entries for Student A can never appear in Student B's retrieval results.
  • Chat sessions are stored per-user and never aggregated in a way that reveals individual student data externally.
  • BYOK API keys are encrypted at rest using AES-256 and never logged in plaintext anywhere in the system.

🛡️ Authentication & Access

  • Supabase Auth handles all authentication: email/password (bcrypt hashed), Google OAuth, and phone OTP (India-optimized).
  • JWT tokens with 1-hour expiry for API authentication. Refresh tokens with 30-day sliding window.
  • Rate limiting: 100 API calls/minute per user for chat endpoints, 30/minute for test submission endpoints, 10/minute for file upload endpoints.
  • Admin access requires 2FA and is audit-logged. All admin actions on student data are recorded with timestamp, admin ID, and action type.

📜 Data Usage Rules

  • Student data is NEVER sold to third parties. Not advertisers, not educational institutions, not government bodies. This is a foundational business rule.
  • Aggregated, anonymized usage statistics may be published (e.g., "DRONA students average 45 minutes of daily study time") for marketing, but individual data is never included.
  • Students can export all their data at any time (GDPR-compliant "Right to Data Portability" even though India hasn't mandated it yet — we do it anyway as best practice).
  • Students can delete their account and all associated data permanently. Deletion is processed within 72 hours and includes all vector embeddings, chat logs, and file uploads.

🔐 AI-Specific Security

  • LLM prompts NEVER include student PII (name, email, phone) in the text sent to external AI providers. Students are referenced by internal ID only.
  • BYOK keys are transmitted to external providers via HTTPS and immediately discarded from memory after the API call completes. They are not included in any logging pipeline.
  • AI responses are filtered for harmful content before delivery. The Head Agent includes a safety layer that rejects responses containing: explicit content, violence, political extremism, or medical/legal advice outside of academic scope.
  • Prompt injection protection: user inputs are sanitized and delimited in the prompt structure. System prompts are not accessible or modifiable by students. Adversarial prompt attempts are logged and flagged.
